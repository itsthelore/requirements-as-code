# RAC v0.8.0 — Explorer

## Problem

RAC currently exposes capabilities through individual CLI commands:

- rac validate
- rac stats
- rac ingest
- rac inspect
- rac improve
- rac relationships
- rac portfolio

These commands are powerful and scriptable, but require users to know:

- Which command to run
- Which file to inspect
- Which output to interpret
- How different capabilities relate to each other

As RAC repositories grow, navigating product knowledge becomes increasingly difficult through command execution alone.

A lightweight exploration experience is required to improve:

- Discoverability
- Adoption
- Day-to-day usability
- Retention

without introducing a web application, desktop application, or repository hosting platform.

---

## Outcomes

Provide a terminal-native exploration experience that allows users to navigate and understand RAC repositories from a single entry point.

Users should be able to answer:

```text
What artifacts exist?
What is missing?
What is connected?
What needs attention?
How healthy is the repository?
```

without needing to remember specific commands.

Explorer shall remain:

- Optional
- Viewer-agnostic
- File-first
- Repository-local

and shall consume existing RAC capabilities rather than implementing new business logic.

---

## Initiatives

### Initiative 1 — Repository Overview

Provide a landing view showing repository-level information.

Example:

```text
Repository Overview

Requirements    42
Decisions       17
Roadmaps         5
Prompts         11
Unknown          2

Portfolio Health 87%
```

Data shall be sourced from:

- rac stats
- rac portfolio

or their underlying service-layer APIs.

---

### Initiative 2 — Artifact Navigation

Allow users to browse artifacts by type.

Example:

```text
Requirements

REQ-001
REQ-002
REQ-003
```

Users shall be able to:

- Navigate using keyboard controls
- Select artifacts
- Open artifact details

---

### Initiative 3 — Artifact Inspection

Display inspection information for selected artifacts.

Example:

```text
REQ-001

Confidence: 100%

Sections

✓ Problem
✓ Requirements
✓ Success Metrics
✗ Risks
✗ Assumptions
```

Data shall be sourced from:

- rac inspect

or equivalent service-layer APIs.

---

### Initiative 4 — Improvement Visibility

Surface improvement opportunities.

Example:

```text
Missing Sections

- Risks
- Assumptions
```

and optionally:

```text
Suggested Template

## Risks

_TODO_

## Assumptions

_TODO_
```

Data shall be sourced from:

- rac improve

or equivalent service-layer APIs.

---

### Initiative 5 — Relationship Navigation

Allow users to view artifact relationships.

Example:

```text
REQ-001

Related Artifacts

ADR-012
ADR-013
ROADMAP-Q3
```

Data shall be sourced from:

- rac relationships

or equivalent service-layer APIs.

---

### Initiative 6 — Portfolio Health Dashboard

Provide a consolidated repository health view.

Example:

```text
Coverage Score      87%

Missing Metrics      4
Unlinked Decisions   2
Unknown Artifacts    1
```

Data shall be sourced from:

- rac portfolio

or equivalent service-layer APIs.

---

### Initiative 7 — Fast Navigation

Support keyboard-driven workflows.

Minimum controls:

```text
↑ ↓   Navigate

Enter Open

Esc   Back

q     Quit
```

Mouse interaction is optional.

---

### Initiative 8 — Repository Search

Provide lightweight search across artifacts.

Users shall be able to:

- Search by ID
- Search by title
- Filter by artifact type

Full-text search is out of scope.

---

## Requirements

### REQ-001 — Terminal Native

Explorer shall operate entirely within a terminal environment.

### REQ-002 — Optional

All RAC capabilities shall remain accessible through standard CLI commands.

Explorer shall not replace existing commands.

### REQ-003 — No Unique Business Logic

Explorer shall consume existing RAC services.

Knowledge intelligence shall not exist exclusively within Explorer.

### REQ-004 — Repository Local

Explorer shall operate on local repositories.

No server component shall be required.

### REQ-005 — Viewer Agnostic Compliance

Explorer shall comply with ADR-014.

Artifacts shall remain usable outside Explorer.

### REQ-006 — Responsive Navigation

Repository navigation shall remain performant for repositories containing at least 1,000 artifacts.

### REQ-007 — Keyboard First

All functionality shall be accessible via keyboard controls.

---

## Non-Goals

### Artifact Editing

Not:

```text
Edit Requirement
Edit Decision
```

Explorer is a viewer, not an editor.

### Repository Hosting

Not:

```text
Remote Repositories
User Accounts
Synchronization
```

### Project Management

Not:

```text
Tasks
Boards
Sprints
Workflows
```

### AI Agents

Not:

```text
Ask Explorer
Generate Requirements
Review Decisions
```

### Web Application

Not:

```text
RAC Cloud
RAC SaaS
RAC Dashboard
```

---

## Success Measures

### Adoption

Users launch:

```bash
rac explorer .
```

as their primary repository entry point.

### Discoverability

New users can navigate a repository without prior knowledge of RAC commands.

### Retention

Explorer becomes the preferred daily interface for repository exploration while CLI commands remain the preferred automation interface.

### Architecture

Explorer remains a consumer of RAC APIs rather than a source of new business logic.

---

## Future Considerations

Future releases may add:

- Relationship graphs
- Artifact lineage views
- Knowledge evolution timelines
- Git history integration
- Plugin support

These capabilities must continue to consume existing RAC services and remain consistent with:

- ADR-012 Repository Intelligence
- ADR-013 Source Control Integration
- ADR-014 Viewer-Agnostic Knowledge
