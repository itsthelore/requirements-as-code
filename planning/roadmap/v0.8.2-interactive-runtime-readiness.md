# RAC v0.8.2 — Interactive Runtime Readiness

## Problem

CLI workflows are short-lived.

Interactive consumers such as Explorer are long-lived.

They require RAC operations to support:

- responsiveness
- progress reporting
- cancellation
- repeated queries

without blocking the interface.

---

## Outcomes

Prepare RAC Core for interactive consumers.

---

# Initiatives

## Initiative 1 — Operation Interface

Represent long-running work consistently.

Examples:

- repository scan
- validation
- ingestion
- relationship analysis

---

## Initiative 2 — Progress Reporting

Operations may expose progress callbacks.

Example:

```text
Scanning artifacts...

Building relationships...

Validating...
```

---

## Initiative 3 — Cancellation Support

Long-running operations should support cancellation where practical.

---

## Initiative 4 — Incremental Refresh

Support refreshing repository state without full reload where possible.

---

## Initiative 5 — Performance Testing

Validate repositories containing:

```text
1000+ artifacts
```

---

# Requirements

## REQ-001 — UI Safe

Operations shall be suitable for async consumers.

---

## REQ-002 — No Event Loop Blocking Assumptions

Core shall not assume synchronous terminal execution.

---

## REQ-003 — Large Repository Support

Repository operations shall remain performant at scale.

---

## Non-Goals

- Textual implementation
- Async rewrite of Core
- Background daemons

---

## Success Measures

Explorer can run RAC operations through workers without freezing.

Large repositories remain usable interactively.
