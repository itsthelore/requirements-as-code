# Agent Decision Enforcement

## Context

The RAC editor extension enforces against the human editing `.rac/` artifacts,
but the AI agent that actually re-litigates settled decisions never sees the
corpus. A design spike (council seats: DevX, Dev Lead, Core, Extension Engineer)
resolved how to get RAC's decisions and enforcement to the agent. This design is
the *how*; the boundary it implements is ADR-067.

## User Need

A developer's agent (Copilot/Cursor/Claude Code) proposes work the team already
ruled out — a banned dependency, a closed architecture call — and a human (often
a junior) rubber-stamps it. The need: the agent should consult and respect the
team's recorded decisions *before* proposing ruled-out work, and the corpus's
structural enforcement should catch contradictions that slip through — across
every agent, with zero per-developer setup.

## Design

Three layers, deterministic and engine-owned (ADR-067):

### 1. Generated agent-context (the wedge)

A new additive CLI verb `rac export --agent-rules [--check]` emits a deterministic
projection of the **live** corpus (Accepted, non-superseded decisions; active
roadmap scope fences; agent-guidance prompts) into per-client files —
`CLAUDE.md`, `AGENTS.md`, `.cursor/rules`, `.github/copilot-instructions.md` —
each in a fenced managed block:

```
<!-- rac:agent-rules:begin sha=<corpus-hash> -->
…distilled bans + closed-decision pointers (id + one-line decision summary)…
<!-- rac:agent-rules:end -->
```

The block is **distilled, not the whole corpus** (it competes for the agent's
attention budget): closed decisions and bans as pointers, plus "how to query the
corpus" (call the `lore` tool / `rac find`). Generation is idempotent and
byte-stable for identical corpus bytes; the hand-authored region outside the
markers is never touched. `--check` exits non-zero when a committed file has
drifted from the corpus, so it can be a CI gate (the same drift discipline as the
vendored Portal shell). Files carry "regenerated by `rac` — edit the corpus, not
this block."

### 2. Live query (MCP + surfaces)

An additive `lore` tool `find_decisions(query, status?)` — `search_index` filtered
to `type=decision`, with a description engineered to fire on "what did we decide /
is X ruled out." The same capability gets a command-palette face ("RAC: What did
we decide about…?") and a CLI face (`rac find --type decision`). It is a *query*,
not a guardrail. The extension registers `lore` per client
(`registerMcpServerDefinitionProvider` for VS Code; a generated `mcp.json` for
Cursor / Claude Code) — consent stays the user's.

### 3. Post-edit guard

The existing cross-artifact diagnostics already fire on agent-written files (a
file an agent edits is just a changed document). On top of that: a reference-site
quick-fix, and an `onWillSaveTextDocument` save-gate that surfaces a contradiction
before the bad state solidifies. The one place a *real* pre-edit veto exists is
Claude Code's `PreToolUse` hook — which the extension **generates** (it cannot be
the interceptor), calling `rac validate -` on the proposed content.

### Per-client feasibility (honest)

| Client | Context (rules) | Live query (MCP) | Pre-edit veto |
| --- | --- | --- | --- |
| Claude Code | yes (`CLAUDE.md`) | yes (native) | **yes** (generated `PreToolUse` hook) |
| Cursor | yes (`.cursor/rules`) | yes (`mcp.json`) | no API → post-edit/save-gate |
| Copilot | yes (`copilot-instructions.md`) | weak/inline none | no API → post-edit/save-gate |

The committed rules files are the universal floor (every client, plus plain
CI/git); MCP and the Claude hook are higher-fidelity layers where the platform
allows.

## Constraints

- Thin client (ADR-063): generation and all corpus logic live in `rac`/`lore`;
  the extension registers, watches, and presents.
- Additive (ADR-007); deterministic, offline, no telemetry.
- No semantic verdict in the engine (ADR-067): the engine retrieves live
  decisions; the agent judges contradiction.
- Generated files are drift-guarded and may not silently rot.

## Rationale

Committed rules files beat an MCP-only design because they reach every agent
(including Copilot) with zero per-developer setup and are reviewable in a PR —
the place adoption actually happens. Keeping the contradiction *judgment* in the
agent keeps `rac` deterministic and trustworthy and avoids the cry-wolf guard
that gets muted. Scoping enforcement to structural facts (a reference to a retired
decision; a banned identifier) is what RAC can assert without lying.

## Alternatives

- Engine-computed semantic contradiction — rejected (ADR-067): non-deterministic,
  needs a model, erodes trust.
- Pre-edit interception in the editor — rejected (ADR-067): no platform API.
- A "corpus chatbot" pull-surface the human must remember to open — rejected: it
  only helps when the human already suspects a decision exists, which is exactly
  when the problem isn't happening.

## Accessibility

The query surface is keyboard-first (command palette / quick-pick). Generated
rules files are plain Markdown, readable and diffable. No surface depends on the
graph or colour.

## Style Guidance

The managed block is terse and scannable — pointers and one-line summaries, not
prose. The setup command ("RAC: Set up agent integration") reports exactly what it
wrote and where, and points the user at the first "ask your agent" moment.

## Open Questions

- Exact per-client rules-file format wrappers and size budget.
- Whether `find_decisions` is a distinct tool or a documented filter on
  `search_artifacts` (ADR-030's additive-tool clause).
- Whether status-aware relationship validation needs any extension beyond the
  existing `relationship-target-superseded` for the PR gate.
