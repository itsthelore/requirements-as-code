# RAC v0.8.4 — Explorer Action Workflows

## Status

Planned

## Context

v0.8.3 presents recommendations, but acting on them still requires users to
leave Explorer, locate files, and open tools by hand.

The Explorer requirement's Act capability requires navigating efficiently
from findings to remediation workflows. Explorer locates, explains, and
recommends; external tools remain responsible for editing, authoring, and
applying changes (DESIGN-action-workflows, ADR-024).

Repositories without structured knowledge also need a supported path in:
the guided import workflow (DESIGN-import-workflow) wraps Core ingestion so
existing documents become structured artifacts without silent mutation.

## Outcomes

- Users move directly from a finding to the affected artifact in their own
  editor.
- Every repository-changing workflow previews its changes and requires
  explicit confirmation.
- Long-running operations report their state, progress, and result.

## Initiatives

### Initiative 1 — Finding Actions

From any finding or recommendation, support:

- Open Artifact
- Open Related Artifact
- Show Recommendation
- Show Relationship Impact
- Open In Editor
- Export Finding

The expected path is: Finding → Recommendation → Open Artifact → Open In
Editor → user remediates.

### Initiative 2 — Editor Integration

Implement editor launch per DESIGN-editor-integration:

- Configurable editor command (for example `explorer.editor.command: cursor`)
  covering editors such as VS Code, Cursor, Windsurf, Vim, Neovim, Emacs, and
  JetBrains IDEs.
- Editor selection offered during first run and changeable later.
- If unconfigured: prompt the user, fall back to the system default, and
  provide guidance.

### Initiative 3 — Guided Import

Implement the import workflow per DESIGN-import-workflow, wrapping the Core
ingest service:

1. Document selection — show detected artifact counts.
2. Import review — show the artifacts that would be created.
3. Confirmation — require explicit user confirmation.

Import is deterministic, with no AI interpretation. RAC Core owns ingestion;
Explorer owns workflow presentation.

### Initiative 4 — Change Preview and Confirmation

Any workflow capable of changing repository contents presents a preview
before applying it. Explorer never silently mutates repository knowledge.

### Initiative 5 — Operation Feedback

Long-running operations display:

- current state
- progress where available
- completion result
- errors

## Non-Goals

- Direct artifact editing or content authoring inside Explorer (ADR-024)
- Saving content changes from Explorer
- Replacing existing editing tools
- Automated fixes applied without confirmation

## Success Measures

- A user can move from a repository finding to the affected artifact open in
  their configured editor without manually locating files.
- No Explorer workflow changes repository contents without a preview and an
  explicit confirmation.
- Import through Explorer produces the same results as `rac ingest`
  (ADR-015).
