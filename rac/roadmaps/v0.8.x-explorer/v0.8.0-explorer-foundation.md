# RAC v0.8.0 — Explorer Foundation

## Status

Planned

## Context

The Explorer requirement defines an interactive product knowledge navigation
environment delivered as a Terminal User Interface (ADR-028).

The v0.7.x series delivered the platform Explorer depends on: a service layer
(`rac.services`), canonical artifact identity (v0.7.11), lookup and resolution
(v0.7.12), and metadata migration (v0.7.13). The `rac.explorer` package exists
as an intentionally empty consumer boundary.

Two gaps remain before an interactive surface can be built:

1. Consumers lack a single navigable representation of a RAC repository.
   Repository discovery should not be duplicated by each consumer.
2. CLI workflows are short-lived; an interactive consumer is long-lived and
   requires responsiveness, progress reporting, and repeated queries without
   blocking the interface.

This milestone closes those gaps and stands up the Explorer application shell.

## Outcomes

- RAC Core exposes a first-class repository model that any consumer can
  navigate without scanning files.
- Core operations are safe for long-lived interactive consumers.
- `rac explorer` launches a Textual-based TUI shell that consumes RAC services
  through a dedicated adapter layer and recovers gracefully from failures.

## Initiatives

### Initiative 1 — Repository Model

Introduce first-class repository objects in RAC Core:

```text
Repository
  contains Artifacts
  connected by Relationships
  described by Diagnostics
```

- A `Repository` object exposes indexed artifacts, validation, relationships,
  and portfolio analysis through the existing services.
- A common artifact representation covers all supported types (requirement,
  decision, roadmap, prompt, design).
- Relationships and diagnostics are shared model objects, reused by
  validation, relationship analysis, and portfolio health.
- Repository models contain no CLI or TUI concepts; consumers shall not
  implement their own repository scanning.

### Initiative 2 — Interactive Runtime Readiness

Prepare Core operations for long-lived consumers:

- Represent long-running work (repository scan, validation, relationship
  analysis, ingestion) through a consistent operation interface.
- Operations may expose progress callbacks and support cancellation where
  practical.
- Support refreshing repository state without a full reload where possible.
- Validate behaviour against repositories containing 1000+ artifacts.

Core shall not assume synchronous terminal execution, and this milestone does
not require an async rewrite of Core.

### Initiative 3 — Textual Application Foundation

Introduce the Explorer TUI application:

```text
RAC Core  →  Explorer Adapter  →  Textual UI
```

Launched as:

```bash
rac explorer
```

### Initiative 4 — Explorer Adapter Layer

Introduce a dedicated boundary between RAC Core and the UI. The adapter
shall:

- invoke RAC services
- translate Core models into UI state
- own no repository intelligence
- prevent widgets from depending on Core internals

### Initiative 5 — Non-Blocking Execution

Explorer shall never execute expensive Core operations directly in UI
handlers. Repository loading, validation, relationship analysis, and
portfolio calculation execute through Textual workers.

### Initiative 6 — Recoverable Errors

Core failures surface as recoverable Explorer states. Failures shall not
crash the interface.

## Constraints

- Explorer consumes Core services only; no business logic exists exclusively
  within Explorer (ADR-015).
- Explorer operates entirely within a terminal, keyboard first, with no mouse
  required (ADR-028, DESIGN-visual-system).
- Meaning never depends on colour alone; limited-colour environments and
  remote sessions are supported (DESIGN-visual-system).
- Explorer includes adapter tests, screen tests, and Core isolation tests,
  organized as service batteries (ADR-027).

## Non-Goals

- Artifact editing or authoring (ADR-024)
- New repository intelligence
- Background daemons or hosted services
- Navigation, health, or recommendation experiences (delivered by v0.8.1+)

## Success Measures

- `rac explorer` opens a repository and renders an application shell without
  blocking on Core operations.
- Explorer navigation in later milestones can be implemented entirely against
  the repository model APIs.
- Explorer remains responsive on repositories of at least 1000 artifacts.
- All existing CLI behaviour and JSON contracts remain unchanged (ADR-007).
