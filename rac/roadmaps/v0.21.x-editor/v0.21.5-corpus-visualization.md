# RAC v0.21.5 — Corpus Visualization (RAC Explorer Webview)

## Status

Planned

## Context

The corpus is a graph of artifacts and relationships, and RAC already has a web
viewer for it: `lore-web` renders the `rac export` payload as a navigable Portal.
This release brings that view into the editor as a side panel, so a maintainer
can see and traverse the graph without leaving VS Code / Cursor — reusing the
existing viewer and the stable export contract (ADR-007, ADR-063) rather than
building a new visualization.

## Outcomes

- The corpus relationship graph is viewable in an editor side panel.
- Selecting an artifact in the panel opens its file; the panel reflects the
  current corpus.
- The view reuses `lore-web` and `rac export`, so it stays consistent with the
  standalone Portal and needs no separate engine.

## Initiatives

### Initiative 1 — RAC Explorer webview

A webview panel that renders the self-contained Portal viewer produced by
`rac export --html --out` — the `lore-web` build with the corpus already
injected (offline, no network). The **RAC: Open Explorer** command builds and
loads it; re-running the command refreshes. Auto-refresh on save is deliberately
not wired, because reloading the webview would reset the viewer's scroll and
selection; an explicit refresh is the cleaner MVP.

### Initiative 2 — Editor integration (deferred)

Two-way sync — open an artifact's file from the graph, reveal the current file's
artifact in the graph — is **deferred**. The Portal is a standalone viewer that
does not post messages to a host, so wiring selections to editor actions would
require changing the `lore-web` viewer build, which is out of scope for this
thin-client release. Recorded here as the natural follow-on.

## Constraints

- Thin client only (ADR-063): the panel consumes `rac export --json`; it does not
  parse or classify artifacts itself.
- Reuse `lore-web`'s viewer build and the export contract (ADR-007); do not fork
  a second viewer.
- The webview is offline and content-free of telemetry, consistent with RAC.

## Non-Goals

- A new visualization engine or graph library distinct from `lore-web`.
- Editing artifacts from the graph (authoring stays in the editor / `rac new`).
- Hosting or publishing the viewer beyond the in-editor panel.

## Implementation Contract

- The webview renders the corpus from the self-contained Portal HTML that
  `rac export --html --out` produces (the `lore-web` viewer with data injected).
- The **RAC: Open Explorer** command builds and loads it; re-running refreshes
  from a fresh export.
- Editor sync (graph selection ↔ open file) is deferred (Initiative 2).

## Success Measures

- The panel renders the corpus relationship graph from `rac export --html`.
- Re-running the command after editing the corpus reflects the change.
- With `rac` absent, the panel shows an actionable message rather than failing.

## Risks

- Coupling the panel to `lore-web`'s build. Mitigated by consuming the stable
  export contract (ADR-007), exactly as the standalone viewer does.
- Export cost on large corpora. Mitigated by on-demand/debounced refresh and
  reusing the cached export where possible.

## Assumptions

- `lore-web`'s viewer can be embedded in a VS Code webview and fed an export
  payload, as it already is for the standalone Portal.
