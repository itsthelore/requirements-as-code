# RAC v0.21.14 — Security & Governance Posture

## Status

Achieved

## Context

RAC's local-only, no-telemetry, offline posture is a strategic asset — it is
what lets a sponsor deploy across regulated teams without a multi-quarter
security review — but today it is a claim, not a provable control a security
office can sign. And enforcement only standardises an organisation when which
findings are blocking is set centrally, not per repository. This release turns
the posture into an attestable control and makes enforcement policy a governed,
central artifact.

## Outcomes

- A documented, verifiable no-egress security posture.
- Central policy-as-config for blocking-versus-advisory enforcement.
- Enforcement that is consistent across repositories.

## Initiatives

### Initiative 1 — A provable posture

A concise security document plus verifiable artifacts: an SBOM, dependency
provenance, and a network-isolation test/attestation demonstrating that the
extension and `rac` run with no egress.

### Initiative 2 — Policy-as-config

A corpus-level policy declaring which finding classes are blocking versus
advisory, consumed by both the editor and the PR gate (v0.21.13).

### Initiative 3 — Fleet readiness

Documentation for standardising the posture and policy across repositories and
teams.

## Constraints

- No telemetry — the posture proves it rather than asserting it.
- Thin client; policy lives in the corpus (governed), not hardcoded.

## Non-Goals

- A hosted control plane, per-user analytics, or certification beyond a
  self-attestable posture.

## Implementation Contract

- A security posture document plus an SBOM, dependency provenance, and a runnable
  no-egress test.
- A corpus policy artifact mapping finding classes to blocking/advisory, consumed
  by the editor and the Action (v0.21.13).

## Success Measures

- The no-egress claim is backed by a runnable isolation test and an SBOM.
- A central policy changes blocking-versus-advisory behaviour in both the editor
  and the PR gate without per-repository edits.

## Risks

- Scope creep toward a control plane. Mitigated by a self-attestable, offline-
  first scope.
- A policy schema is new corpus surface. Kept additive.

## Assumptions

- Dependencies permit an SBOM/provenance, and a policy artifact can be additive
  to the corpus and CLI.
