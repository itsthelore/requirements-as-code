# RAC v0.21.6 — Extension Robustness and Release

## Status

Planned

## Context

By v0.21.5 the extension is feature-complete for a first audience: validation,
enforcement, authoring, navigation, awareness, and a graph view. What remains is
the hardening that separates a working extension from a publishable one — scoped
activation, version-skew handling, caching, real logging — and the packaging to
ship it. This release is the pre-publish pass that makes the extension robust on
a real maintainer's machine and ready for the marketplace.

## Outcomes

- The extension activates only in RAC workspaces, leaving other repositories
  untouched.
- A `rac` version mismatch is detected and surfaced, not silently wrong.
- Hover, completion, and lookups stay responsive on a large corpus by serving
  from an in-memory cache rather than spawning `rac` per interaction.
- Errors go to a dedicated output channel, and the extension is packaged for the
  VS Code Marketplace and OpenVSX.

## Initiatives

### Initiative 1 — Scoped activation

Activate only when the workspace contains a `.rac/config.yaml`, so non-RAC
repositories never load the extension or see RAC diagnostics.

### Initiative 2 — Version-skew detection

Compare the installed `rac` `schema_version` (and version) with what the SDK
expects, and warn on a mismatch with actionable guidance, rather than risking a
misread contract.

### Initiative 3 — Caching and performance

An in-memory cache of resolve/find/index results, invalidated on document and
corpus change, so hover, completion, references, and symbols do not spawn `rac`
on every interaction. The escape hatch if the subprocess model still strains is a
long-lived language server, noted but out of scope here.

### Initiative 4 — Logging and packaging

A dedicated "RAC" output channel for errors (replacing console logging, no
telemetry), and Marketplace / OpenVSX packaging metadata — `repository`, `bugs`,
`keywords`, categories, the CHANGELOG, and the extension **icon** (the RAC
lore-explorer mascot). The actual publish (an `vsce`/`ovsx` step requiring a
publisher account) remains the one manual human step, called out rather than
automated here.

## Constraints

- Thin client only (ADR-063): caching memoizes `rac` outputs; it does not become
  a second engine.
- No telemetry; logging is local to the output channel, consistent with RAC.
- Caches are invalidated on change so the extension never shows stale findings.

## Non-Goals

- A long-lived LSP server (a possible later refactor; not required to ship).
- A native in-process engine or web-VS-Code support (ADR-063).
- New analysis features beyond v0.21.0–v0.21.5; this release is hardening + release.

## Implementation Contract

- The extension activates only with a `.rac/config.yaml` present.
- A `rac` version/`schema_version` mismatch produces an actionable warning.
- Lookups are served from an invalidated-on-change cache; correctness is never
  traded for the cache (stale entries are dropped on change).
- Errors are written to a "RAC" output channel; the extension ships to the
  Marketplace and OpenVSX.

## Success Measures

- Opening a non-RAC repository does not activate the extension.
- A skewed `rac` version surfaces a warning.
- Repeated hover/completion on a large corpus is served from cache without a new
  `rac` process each time; edits still invalidate stale results.
- The extension installs from the Marketplace / OpenVSX and logs to its channel.

## Risks

- Cache staleness causing wrong findings. Mitigated by invalidating on document
  and corpus change and preferring a miss over a stale hit.
- Subprocess model hitting a ceiling on very large corpora. Mitigated by caching;
  a language server remains the documented escape hatch.

## Assumptions

- `.rac/config.yaml` is a reliable signal of a RAC workspace.
- `rac --version` / a `schema_version` field is available to compare against.
