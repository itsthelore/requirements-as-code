# RAC v0.26.0 ‚Äî Appearance

## Status

Planned

## Context

v0.25.0 made the corpus **reachable**; v0.26.0 makes the Explorer **comfortable
to read**. The Explorer ships one curated theme ‚Äî `rac-lantern`, warm amber on
near-black ‚Äî as its default, with a `theme` preference
(`src/rac/explorer/preferences.py`) that already accepts any registered Textual
theme and a `/settings` cycle that enumerates `app.available_themes`
(`src/rac/explorer/widgets/views.py`). ADR-028 fixes the principle that makes a
palette swap safe: meaning never rides on colour alone ‚Äî every type tag, key
chip, and status icon carries text, so disabling or changing the palette loses
no information.

A design pass over the Explorer's visual language (`explorer-visual-system`,
RAC-KTQ63DT43E1Y) surfaced demand for a **light reading mode**: the same frame,
the same information, rendered dark-ink-on-paper for users on light terminals or
who want a calmer reading surface. A runtime proof confirmed a light Textual
theme recolours the entire UI with no widget or stylesheet change ‚Äî the whole
interface derives its colour from theme tokens. This release turns that proof
into a shipped, curated choice and hardens the stylesheet so every theme ‚Äî light
included ‚Äî recolours cleanly.

The scope is the **terminal** surface only. The bolder web explorations' glow,
shadows, gradients, and serif type are browser capabilities the terminal does
not have; the dark default already carries the "bold" palette, so the only
genuinely new rendering choice is the light theme.

## Outcomes

User-felt outcomes (the release narrative leads only with these):

- **A curated light theme** ‚Äî `rac explorer` offers a first-class light theme
  (`rac-parchment`: warm paper, dark ink, a deepened lantern amber that stays
  legible on light) selectable in `/settings` beside the dark default and
  persisted through the existing `theme` preference. Users on light terminals
  get an on-brand, legible surface instead of an off-the-shelf Textual theme.
- **Theme-clean rendering** ‚Äî every status, type, and validation signal stays
  legible under any registered theme, because the stylesheet derives all colour
  from theme tokens and meaning is carried by text, not hue (ADR-028). The light
  theme is the forcing function that proves it.
- **A discoverable choice** ‚Äî the `theme` preference and the curated RAC themes
  are documented, so users can find and set them.

## Initiatives

### Tier 1 ‚Äî this is the release

- **WS1 ‚Äî Curated light theme** `[user-facing]`. Register `rac-parchment`
  (`dark=False`; warm paper background, dark-ink foreground, a deepened amber
  accent legible on light, light surfaces) in `src/rac/explorer/theme.py` and on
  app mount, so it joins `rac-lantern` as a curated RAC theme and appears
  automatically in the `/settings` theme cycle (which already enumerates
  `app.available_themes`). Selection persists through the existing `theme`
  preference; an unknown or removed theme name still falls back to the curated
  default at startup. Requirement `rac-explorer-light-theme` (minted during
  implementation).
- **WS2 ‚Äî Theme-clean stylesheet audit** `[internal]`. Audit `explorer.tcss` and
  the widgets for any hard-coded colour that does not derive from a theme token
  (`$accent` / `$surface` / `$panel` / `$text` / `$text-muted` / the state
  colours), so the light theme ‚Äî and any theme ‚Äî recolours without a stranded
  dark value. Confirm type tags, status chips, the mascot, and scrollbars stay
  legible on light; the type-tag hues already carry text, so this is about
  contrast, not meaning.

### Tier 2 ‚Äî keep, scope down (cut first if the release runs short)

- **WS3 ‚Äî Theme documentation** `[user-facing]`. Document the `theme` preference
  and the curated RAC themes (`rac-lantern` dark, `rac-parchment` light) in the
  Explorer docs, including how to set a theme in `/settings` and that all meaning
  survives any palette (ADR-028).

### Tier 3 ‚Äî deferred / documented (not built this release)

- **Web-only glamour** ‚Äî the bolder web mockups' glow, drop shadows, gradients,
  and serif / proportional type are browser capabilities the terminal does not
  have; the dark default already delivers the bold palette. Out of scope,
  permanently, for the TUI (Non-Goals).
- **A high-contrast / accessibility theme variant** ‚Äî a candidate for a later
  v0.26.x item once the light theme has landed.
- **A knowledge-graph view** ‚Äî surfaced during the same exploration and recorded
  under `explorer-knowledge-graph`; it is its own feature, not appearance.

## Success Measures

- `rac-parchment` is selectable in `/settings`, renders the real TUI legibly in
  light, and persists across sessions through the `theme` preference.
- With colour disabled or under the light theme, every validation, status, and
  type signal remains readable ‚Äî no colour-only meaning (ADR-028).
- An unknown or removed theme name in the preference still falls back to the
  curated default without breaking startup.
- No change to the JSON contract, the MCP surface, or any write capability; the
  addition is a registered theme plus an existing preference value only
  (ADR-007).

## Constraints

Non-negotiable guardrails, each a recorded decision:

- Colour communicates focus, not meaning; every theme keeps type tags, key
  chips, and status icons carrying text (ADR-028).
- Preferences stay optional and non-blocking; a missing or unknown theme falls
  back to the curated default and never blocks startup (ADR-028).
- The addition is additive only ‚Äî a registered theme and an existing preference
  value; no JSON-contract, MCP, or write change (ADR-007).
- 256-colour terminal compatibility; no colour-only meaning (ADR-028).
- Roadmap lifecycle follows ADR-061; this item stays `Planned` until delivered.

## Non-Goals

- **No serif or proportional fonts** ‚Äî the terminal is monospace; the web
  mockup's serif body is not portable.
- **No glow, drop shadow, gradient, or blur** ‚Äî terminal rendering has no such
  primitives; the dark default already carries the bold palette.
- **No theme-authoring UI** ‚Äî themes ship in code and are chosen by name.
- **No artifact-content or syntax theming** beyond Textual's Markdown rendering.

## Risks

- **The light theme exposes a hard-coded colour that does not follow the theme.**
  Mitigation: WS2 audits the stylesheet and widgets for stranded values before
  WS1 is considered done.
- **The `/settings` cycle over `available_themes` is long** (it already lists the
  built-in Textual themes), so a curated RAC light theme can be lost among them.
  Mitigation: name it `rac-parchment` so the two RAC themes sort together under
  the `rac-` prefix; broader theme-list curation is a deferred follow-up.
- **Low-contrast amber on light paper harms legibility.** Mitigation: the accent
  deepens on light (proven at runtime) rather than reusing the dark theme's
  bright amber.

## Assumptions

- The `theme` preference and the `/settings` cycle over `app.available_themes`
  already exist, so the light theme needs no new selection or persistence
  machinery ‚Äî only a registered `Theme`.
- ADR-028's "meaning survives any theme" already holds in the widgets (type tags
  and chips carry text), so a palette swap is safe by construction.
- The parchment palette proven at runtime is the WS1 starting point.
