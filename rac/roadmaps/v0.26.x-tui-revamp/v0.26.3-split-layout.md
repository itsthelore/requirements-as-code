# RAC v0.26.3 ÔÇö Split Layout

## Status

Planned

## Context

The Explorer's frame swaps one view into the context region at a time; browsing
and reading alternate. The split-view exploration kept a dense list on the left
and a persistent reading pane on the right (master-detail), so a user moves
through the list while the selected artifact stays open ÔÇö the pattern that suits
review and cross-referencing. This release adds a split workspace layout as an
alternative to the tree + swap frame, building on the portfolio list (v0.26.2)
as the master pane.

## Outcomes

- **A master-detail layout** ÔÇö an optional split frame: the portfolio list on
  the left, the artifact reading pane on the right, selection in the list
  updating the pane in place, so reading and scanning happen together.
- **A layout preference** ÔÇö the split layout is chosen and persisted like the
  theme, so users keep the frame that fits their work.

## Initiatives

### Tier 1

- **WS1 ÔÇö Split frame** `[user-facing]`. A master-detail layout reusing the
  v0.26.2 list as the master pane and the existing `ContextView` as the detail
  pane; arrow keys move the selection, the pane follows, Tab crosses panes ÔÇö
  rendered from loaded state only (ADR-015).
- **WS2 ÔÇö Layout preference** `[user-facing]`. A `layout` preference (`frame`
  default, `split` alternative) persisted to `explorer.json` and selectable in
  `/settings`.

### Tier 2

- **WS3 ÔÇö Responsive fallback** `[internal]`. Below a width threshold the split
  collapses to the single-pane frame, mirroring how the sidebar already hides
  under 80 columns.

Notes from implementation:

- The split is **composition over existing widgets** (ADR-015): the master is the
  v0.26.2 `PortfolioView`, the detail the existing `ContextView`; no logic moves
  into the layout. The `NavigationSidebar` is always composed (it hosts the
  loaded browser state and reveal/status) and simply hides in split.
- **Live switch** re-mounts the `#workspace` for the new preference, then reloads
  ÔÇö an explicit preference change, not a navigation jump, so the "one stable
  frame" rule (ADR-028) holds within each layout.

## Success Measures

- In split layout, moving the list selection updates the reading pane in place
  without a view swap, and the layout persists across sessions.
- The single-frame layout is unchanged at the default preference; no regression
  to the existing navigation.

## Risks

- Two persistent panes strain narrow terminals. Mitigation: WS3's width-threshold
  fallback to single-pane, mirroring the sidebar's existing behaviour.
- The split layout depends on v0.26.2's list. Mitigation: sequence it after the
  portfolio list; if v0.26.2 slips, this slips with it.

## Assumptions

- The portfolio list (v0.26.2) provides the master pane, and `ContextView`
  already renders the detail, so the layout is composition over existing widgets.
