# RAC v0.26.2 — Portfolio List

## Status

Planned

## Context

The Explorer browses through a tree sidebar and reads one artifact at a time.
For a corpus of hundreds of artifacts, users also need to *scan across* the
portfolio — triage by status, filter to what needs attention, sort by recency.
The dense "list" exploration (the Linear-style mockup) showed a metadata-column
view — type, ID, title, status, links, recency — that answers "what is the
state of everything?" in one screen. This release adds that as a real view,
reusing the loaded repository state and the review findings; it never calls Core
directly (ADR-015).

## Outcomes

- **A portfolio list view** — a dense, scannable list of artifacts with type
  tag, ID, title, status chip, link count, and recency, opened with a command
  (e.g. `/list`) or a view toggle, every row keyboard-navigable into the
  artifact.
- **Triage filters** — filter the list to a status (accepted / proposed /
  needs-attention / invalid) and sort by recency, so a maintainer finds the work
  without walking the tree.

## Initiatives

### Tier 1

- **WS1 — List view** `[user-facing]`. A columns view over the loaded
  `BrowserState` and the review findings: type, id, title, status, links,
  recency; rows render from already-loaded state (ADR-015), open on Enter, and
  reveal in the sidebar.
- **WS2 — Filters and sort** `[user-facing]`. Status filter tabs and a recency
  sort, reusing the git-derived recency signal (ADR-045).

### Tier 2

- **WS3 — Saved views** `[internal]`. Persist a couple of named filters
  (needs-attention, invalid) to the preferences file.

### Tier 3 — deferred

- Per-type custom columns beyond the shared metadata stay out of scope; the
  view shows the common columns every artifact has.

## Success Measures

- `/list` (or the toggle) shows every artifact with status and recency,
  filterable to needs-attention, each row opening the artifact.
- The view derives entirely from loaded state and the review service — no new
  Core call and no new JSON contract (ADR-015, ADR-007).

## Risks

- Recency needs per-artifact git data that can be slow on large repos.
  Mitigation: reuse the existing recency derivation (ADR-045) and compute lazily.

## Assumptions

- `BrowserState`, the review findings, and the git-derived recency already exist
  in the adapter, so the view is presentation over current state.
