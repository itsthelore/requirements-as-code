# RAC v0.26.1 — Legibility

## Status

Planned

## Context

v0.26.0 shipped the light theme and confirmed the stylesheet recolours entirely
from theme tokens. One hard-coded colour set remains: the artifact type-tag hues
(`REQ` / `ADR` / `RMP` / `PRM` / `DSG`) live in
`src/rac/explorer/widgets/sidebar.py` and are shared across themes. They carry
text, so meaning is never lost (ADR-028) — but on the light `rac-parchment`
background some hues (cyan `PRM` in particular) read at lower contrast than on
near-black. This release makes the tag hues theme-aware so they stay crisp on any
background, and adds a high-contrast theme for users who need maximum legibility.

## Outcomes

- **Legible tags on any theme** — type-tag hues adapt to the active theme (a
  light-tuned set on light backgrounds, the current set on dark) so every tag
  reads crisply, without changing what the tag means: it is still text + colour.
- **A high-contrast theme** — a curated `rac-high-contrast` theme joins the pair,
  maximising foreground/background separation for low-vision and
  bright-environment use, selectable through the existing `theme` preference.

## Initiatives

### Tier 1

- **WS1 — Theme-aware type tags** `[user-facing]`. Resolve the tag hue from the
  active theme rather than a fixed map, with a light-background variant tuned for
  contrast on `rac-parchment`; the tag text is unchanged, so meaning is untouched
  (ADR-028).
- **WS2 — High-contrast theme** `[user-facing]`. Register `rac-high-contrast`
  (maximal ink/paper separation, the amber accent retained) so it joins the
  `/settings` cycle beside the curated pair.

### Tier 2

- **WS3 — Documentation** `[docs]`. Note the high-contrast theme and the contrast
  guarantee in the Explorer docs.

## Success Measures

- Every type tag reads legibly against both curated themes' panel backgrounds,
  verified by a render and a token check.
- `rac-high-contrast` is selectable and persists; no widget loses meaning when
  colour is removed (ADR-028).

## Risks

- A theme-aware hue map drifts from the design system's type-tag identity.
  Mitigation: keep the same five hue *families*, adjusting only lightness for
  contrast; the canonical hues stay recorded in `explorer-visual-system`.

## Assumptions

- The tag-rendering path (`type_tag` in `sidebar.py`, reused by the palette) is
  the single place hues are chosen, so theme-awareness is one change.
