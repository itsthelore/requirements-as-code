# lore-verify v0.2.0 — Breadth

## Outcomes

- A compiled test runs unchanged across **multiple targets** (at least dev and
  production) and **multiple operating systems** (at least one non-host OS), with
  target and OS injected at run time (LV-ADR-002), proving the target- and
  OS-agnostic property the prototype only demonstrated on the host.
- Tests run **fast and in parallel** (sharded), so breadth does not cost
  proportional wall-clock — the Run profile stays distinct from the slow Drive
  profile (`drive-compile-run-architecture`).
- A single Drive session can produce and verify **multiple capabilities**, opening
  one proposal PR that adds several `## Verified By` references at once.
- The **trace artifact is the review surface**: a reviewer confirms the run from
  the trace attached to the PR, with no local execution.
- All of the above runs on the **local runner only**; the hosted VM-fabric runner
  remains a separate track (RAC ADR-083, `lore-verify-programme` Initiative 6).

## Initiatives

### Initiative 1 — Multi-target execution

Make a compiled test runnable against any configured target via injected `baseURL`
+ auth strategy (LV-ADR-002). Handle the divergence the prototype could ignore:
dev can be seeded; production cannot, so production is verified with read-only /
idempotent test variants. Targets are declared in config, not in tests.

### Initiative 2 — Multi-OS matrix

Run compiled tests across operating systems (Mac / Windows / Linux) on the local
runner, via the matrix the runner interface exposes. The agent (Drive) still runs
once on the host; only the compiled tests fan out across the matrix.

### Initiative 3 — Parallel, sharded runs

Execute the test suite in parallel with sharding so breadth scales sub-linearly in
wall-clock. Determinism of each compiled test is the precondition (the fidelity
gate from `faithful-session-to-test` already enforces it).

### Initiative 4 — Multi-capability sessions and batched write-back

Let one Drive session verify several capabilities and emit a single proposal PR
adding multiple `## Verified By` references, each pointing at its compiled test —
still proposal-only, human-reviewed (RAC ADR-065, LV-ADR-001).

### Initiative 5 — Trace as the PR review surface

Attach the replayable trace for each accepted test to the proposal PR so the test,
its assertions, and its evidence are reviewable together without a local run
(`faithful-session-to-test` REQ-004).

## Success Measures

- One compiled test passes against both a dev and a production target and on at
  least one non-host OS, with no edit to the test.
- A multi-capability session opens one PR that, once merged, clears several
  `unverified-capability` gaps in `rac coverage` at once.
- The parallel suite's wall-clock is materially below the serial sum, and every
  run remains green and stable (no flake introduced by parallelism).
- A reviewer signs off using only the PR diff and the attached traces.

## Assumptions

- v0.1.0 closed the single-target/single-OS/single-capability loop, so this
  release is about breadth, not first principles.
- Targets can be modelled as config (baseURL + auth + seed strategy); production
  read-only variants are sufficient to verify without mutating prod.
- The local runner can drive a multi-OS matrix (containers / local VMs);
  cloud-owned OSes remain the hosted track's concern.

## Risks

- **Prod verification mutates production.** Mitigation: production targets use
  read-only / idempotent variants only (Initiative 1); a target declares whether
  it is seedable.
- **Parallelism reintroduces flake.** Mitigation: per-test determinism is enforced
  upstream by the fidelity gate; sharding isolates state per worker.
- **Breadth before the loop is solid.** Mitigation: this is explicitly v0.2.0,
  after v0.1.0 proves the end-to-end loop on one case.
- **Multi-OS surface balloons scope toward the hosted runner.** Mitigation: the
  hosted VM fabric stays out of scope here (RAC ADR-083); v0.2.0 uses the local
  runner only.
